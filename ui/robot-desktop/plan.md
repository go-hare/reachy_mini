# Robot Desktop 阶段计划

## 当前理解

目标不是继续沿用 Codex TUI，也不是做一个只包网页的外壳。

本阶段要把 `ui/robot-desktop/` 收敛成一个真正可运行的桌面客户端基线，当前选型为 **ClawX**，用途是：

- 先作为 Reachy Mini 机器人平台的桌面端 UI 壳
- 先跑通页面与桌面应用启动链路
- 暂时接受后端未接入，只要能看到桌面 UI
- 下一阶段再把默认的 OpenClaw / 内置 agent 能力逐步替换成我们的 Python 机器人服务

## 本阶段技术方案

1. 清理之前 Open WebUI Desktop 遗留的进程、`node_modules` 和构建产物，避免污染当前基线。
2. 保持 `ui/robot-desktop/` 以 ClawX 源码为准，确保目录干净可重复安装。
3. 使用 ClawX 原生的 Electron + React + Vite 开发链路完成首次安装与编译。
4. 先验证“桌面客户端能打开并看到页面”，不以联通机器人后端为本阶段完成条件。
5. 基线可运行后，再开始讨论功能裁剪：
   - 删掉与 OpenClaw 强绑定但对机器人平台无关的页面或能力
   - 保留聊天、会话、Agent 管理、多任务入口等对机器人平台有价值的外壳
   - 规划把通信层替换为 Python WebSocket / HTTP 服务

## 当前已确认的答案

### 1. UI 形态

答案：要桌面客户端，不要 TUI，不要只启动命令行。

### 2. 代码放置位置

答案：放在当前仓库内，路径固定为 `ui/robot-desktop/`，作为项目内置 UI。

### 3. 本阶段优先级

答案：先看得到页面，再决定裁剪哪些模块。

### 4. 后端策略

答案：后续将默认 agent/runtime 替换为 Python 机器人服务；当前阶段先不强求打通后端。

## 当前假设

- 当前用户主要在 Windows 环境开发。
- `ui/robot-desktop/` 可以独立作为一个前端/桌面子工程维护。
- 允许保留 ClawX 当前页面结构作为过渡，后续再做机器人平台化裁剪。

## 本阶段完成标准

- `ui/robot-desktop/` 中不存在旧 UI 基线残留依赖污染。
- 依赖可成功安装。
- 至少能完成一次开发启动或构建验证。
- 能明确告诉用户“当前桌面页面是否已可看、卡在什么点、下一步该删什么”。
